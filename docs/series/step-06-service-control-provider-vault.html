Step 06

Service control과 provider vault로 키 관리 구조를 바꾼 이유

거래소 키를 GitHub Secrets에 직접 두지 않고 서비스 콘솔과 Supabase vault로 옮긴 이유를 설명합니다.

게시일 2026-03-15 수정일 2026-03-15 키워드 서비스 콘솔, provider vault, API 키 관리, Supabase, Bybit, Binance

왜 GitHub Secrets에 거래소 키를 직접 두지 않았나

개인 자동화 스크립트라면 GitHub Secrets에 거래소 키를 넣고 배치를 돌리는 방식도 가능합니다. 하지만 서비스형 구조로 바꾸려면 운영자가 웹 화면에서 설정을 바꾸고, 그 결과가 저장소를 거쳐 다음 배치에 반영되는 흐름을 갖는 편이 훨씬 자연스럽습니다.

또한 거래소 키를 코드 저장소와 너무 강하게 묶어 두면, 설정 변경과 운영 이력이 분리되지 않습니다. 그래서 현재 구조에서는 GitHub Actions에는 공용 인프라 키만 두고, 거래소 키는 서비스 콘솔을 통해 Supabase vault에 암호화 저장하도록 바꿨습니다.

provider를 왜 세 카드로 나눴나

Bybit는 선물 실행 계정, Binance는 실시간 시세, CoinGecko는 메타와 유니버스 보강 소스로 각각 역할이 다릅니다. 같은 화면 안에 있어도 입력 필드와 설명을 분리해야 운영자가 헷갈리지 않습니다. 어느 키가 주문용이고 어느 키가 데이터용인지 구분이 안 되면, 실제 운영에서 실수가 더 많이 납니다.

그래서 Service control은 단순한 설정 폼이 아니라, provider의 역할을 설명하는 콘솔이 되도록 설계했습니다. 화면에서 역할과 업데이트 시각, 키 힌트가 함께 보이는 이유도 같은 맥락입니다.

execution target과 arm을 분리한 이유

Bybit 키를 저장했다고 해서 바로 live가 켜지는 구조는 위험합니다. 키 저장은 단지 configured 상태일 뿐이고, execution target, live execution flag, live crypto enable, arm 단계를 별도로 통과해야만 future live 준비 상태가 되도록 가드를 나눴습니다.

이렇게 하면 설정 실수 하나가 곧바로 주문 실행으로 이어지는 일을 막을 수 있습니다. 서비스형 자동매매에서 가장 중요한 것은 편의보다 단계적 승인 흐름입니다.

서비스 콘솔이 중요한 이유

Service control은 예쁜 설정 페이지가 아니라, 프로젝트의 실행 경로를 하나의 화면에서 정리하는 운영 입구입니다. provider 키, execution target, runtime profile, 하드 리셋이 모두 한 화면에 있지만, 각각의 책임을 섞지 않도록 문구와 섹션을 나눠 두었습니다. 결국 콘솔이 있어야 운영자가 어떤 키를 저장했고, 어떤 전략 규칙을 적용하고 있으며, 다음 배치에서 무엇이 바뀔지 스스로 설명할 수 있습니다.

키 관리 구조를 바꾸면 생기는 운영 안정성

거래소 키를 GitHub Secrets에만 두는 구조는 개인 자동화에는 편하지만, 서비스 콘솔을 중심으로 운영 흐름을 만들기에는 한계가 있습니다. 키를 교체하거나 제공자 구성을 바꿀 때마다 저장소 설정과 배치 설정을 함께 만져야 하기 때문입니다. 그럴수록 운영자는 설정 경로를 기억해야 하고, 실수 가능성도 커집니다.

provider vault를 둔 이유는 키 저장 위치를 한 곳으로 통합하기 위해서입니다. 운영자는 설정 화면에서 저장하고, 배치는 Supabase vault에서 읽고, GitHub에는 공유 인프라 시크릿만 남깁니다. 이렇게 분리하면 관리 책임이 명확해지고, 키를 바꾸는 작업도 운영 콘솔 안에서 끝낼 수 있습니다.
