Step 07

개요·모델 성과·포지션·설정으로 UI를 분리한 운영자 경험 설계

한 화면에 모든 것을 넣는 대신 운영자 관점에 맞춰 UI를 분리한 이유와 각 화면의 책임을 설명합니다.

게시일 2026-03-15 수정일 2026-03-15 키워드 운영자 UI, 대시보드 정보구조, 모델 성과, 포지션 화면, Service control

한 화면에 모든 것을 넣었을 때의 문제

자동매매 UI는 데이터가 많을수록 좋아 보이기 쉽습니다. 하지만 실제 운영에서는 포지션을 보고 싶은 순간과 설정을 바꾸고 싶은 순간, 모델별 성과를 비교하고 싶은 순간이 다릅니다. 그 모든 순간을 한 화면에 섞어 놓으면 오히려 판단 피로가 빠르게 올라갑니다.

이번 UI 재구성의 핵심은 보기 좋게 꾸미는 것이 아니라, 운영자의 질문 순서에 맞게 화면을 나누는 것이었습니다. 지금 시스템이 살아 있나, 어떤 모델이 잘하고 있나, 어떤 포지션이 열려 있나, 무엇을 바꿀 수 있나를 각각 다른 화면에 두었습니다.

개요 화면의 역할

개요 화면은 오늘의 전체 그림만 보여줍니다. heartbeat, 최근 손익 합계, 오픈 포지션 수, 최근 사이클 신호 수처럼 지금 상황을 빠르게 읽는 데 필요한 값만 남겼습니다. 긴 표나 설정 폼은 이 화면에서 모두 제거했습니다.

이렇게 해야 운영자가 브라우저를 열었을 때 지금 시스템이 멈췄는지, 배치가 돌아가고 있는지, 대체로 손익 흐름이 어떤지만 즉시 파악할 수 있습니다. 개요는 보고 판단하는 화면이지, 입력하는 화면이 아닙니다.

모델 성과와 포지션을 탭으로 나눈 이유

모델 성과 화면은 A/B/C/D 모델을 하나의 표에 몰아넣지 않고 탭으로 나눴습니다. 포지션 화면도 마찬가지입니다. 모델 컬럼을 표 한가운데 계속 읽는 것보다, 아예 모델 단위로 분리해서 보는 편이 훨씬 명확하기 때문입니다.

특히 포지션 화면은 현재 포지션, 최신 setup, 최근 체결 로그를 모델별로 묶어 보는 것이 중요했습니다. 어떤 모델이 어떤 entry / SL / TP 계획을 냈고, 최근 체결이 어떻게 기록되었는지가 한눈에 이어져야 하기 때문입니다.

설정 화면만 입력을 받는 구조

입력 폼은 `/settings`에만 남기고 나머지 화면에서는 제거했습니다. 이 결정은 단순한 미관 문제가 아니라 안전장치입니다. 개요나 포지션을 보던 중에 실수로 설정을 바꾸는 일이 없도록, 입력은 의도적으로 설정 화면에 들어가서만 하게 만든 것입니다. 운영자 경험은 편리함과 안전성의 균형이며, 지금 구조는 잘못된 맥락에서 설정이 바뀌지 않도록 하는 쪽을 선택했습니다.

정보구조를 나누면 무엇이 쉬워지는가

운영자는 한 번에 모든 데이터를 보고 싶어 하는 것처럼 보여도, 실제로는 목적별로 다른 질문을 합니다. 시스템이 살아 있나, 모델 성과가 어떤가, 현재 포지션은 어떤가, 설정은 무엇으로 되어 있나를 같은 시선으로 읽지 않습니다. 그래서 개요, 모델 성과, 포지션, 설정으로 나눈 화면 구조는 보기 좋은 디자인 선택이라기보다 질문의 종류를 분리한 설계입니다.

이렇게 나누면 화면 하나하나의 밀도를 올리기 쉬워지고, 빈 상태를 표현하기도 쉬워집니다. 개요는 상태 파악에 집중하고, 모델 페이지는 비교에 집중하고, 포지션 페이지는 체결과 종료 로그에 집중하고, 설정 페이지는 입력만 다룹니다. 이 단순한 원칙 하나가 서비스 인상을 크게 바꿉니다.

운영자에게 중요한 것은 정보량보다 맥락이다

UI를 나누면 한 화면의 정보량은 줄어들 수 있지만, 대신 한 번에 판단해야 하는 질문의 종류가 줄어듭니다. 이 차이는 생각보다 큽니다. 운영자는 적은 데이터를 보는 것이 아니라, 지금 해야 할 판단에 맞는 데이터만 보게 됩니다. 서비스처럼 보인다는 말은 결국 이 맥락 분리에서 시작됩니다.
