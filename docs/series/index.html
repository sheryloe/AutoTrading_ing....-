AI_Auto 자동매매 리빌드 10단계

AI_Auto를 단순한 자동매매 스크립트에서 서비스형 선물 데모 콘솔로 재구성한 과정을 10단계로 나눠 정리했습니다. 운영 리스크, provider vault, 8분 배치, intrabar 체결, 하드 리셋, autotune, 라이브 전환 체크리스트까지 한 흐름으로 이어집니다.

Step 01 자동매매를 기능이 아니라 운영 리스크로 다시 정의한 이유

AI_Auto 프로젝트를 기능 추가 중심이 아니라 운영 리스크 관리 중심으로 다시 정의한 배경과 기준을 정리합니다.

 Step 02 Vercel·Supabase·GitHub Actions로 서비스형 구조를 다시 잡은 과정

AI_Auto를 단일 Flask 앱에서 서비스형 운영 구조로 재설계한 이유와 Vercel, Supabase, GitHub Actions의 역할을 설명합니다.

 Step 03 Top 5 메이저 코인과 선물 유니버스로 범위를 줄인 이유

밈 코인을 제외하고 Top 5 메이저 코인 futures 데모로 범위를 고정한 배경과 provider 소스 분리 이유를 설명합니다.

 Step 04 4개 모델이 entry·TP·SL을 직접 제안하는 planner 구조로 바꾼 이유

단순 점수형 모델에서 진입가, 손절가, 목표가를 제안하는 planner 모델 구조로 바꾼 이유와 모델별 역할을 설명합니다.

 Step 05 8분 배치와 1분 intrabar 체결 시뮬레이션을 함께 쓴 이유

8분 주기 배치 구조에서 entry, TP, SL이 중간에 터치되었는지 놓치지 않기 위해 1분 intrabar 시뮬레이션을 넣은 이유를 설명합니다.

 Step 06 Service control과 provider vault로 키 관리 구조를 바꾼 이유

거래소 키를 GitHub Secrets에 직접 두지 않고 서비스 콘솔과 Supabase vault로 옮긴 이유를 설명합니다.

 Step 07 개요·모델 성과·포지션·설정으로 UI를 분리한 운영자 경험 설계

한 화면에 모든 것을 넣는 대신 운영자 관점에 맞춰 UI를 분리한 이유와 각 화면의 책임을 설명합니다.

 Step 08 일별 PnL 기록과 주간 autotune을 운영 루프에 넣은 이유

신호 생성만이 아니라 결과 기록과 주간 autotune까지 운영 루프로 묶은 이유를 설명합니다.

 Step 09 하드 리셋과 시드 정책을 분리해 선물 데모를 다시 시작한 이유

런타임 저장과 하드 리셋을 분리하고, 10000 USDT 시드로 futures demo를 다시 시작하게 만든 이유를 설명합니다.

 Step 10 실거래로 가기 전에 꼭 확인해야 할 라이브 전환 체크리스트

현재 서비스형 futures demo 구조에서 실거래로 넘어가기 전에 무엇을 더 검증하고 고정해야 하는지 정리합니다.
