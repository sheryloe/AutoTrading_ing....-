Step 02

Vercel·Supabase·GitHub Actions로 서비스형 구조를 다시 잡은 과정

AI_Auto를 단일 Flask 앱에서 서비스형 운영 구조로 재설계한 이유와 Vercel, Supabase, GitHub Actions의 역할을 설명합니다.

게시일 2026-03-15 수정일 2026-03-15 키워드 Vercel, Supabase, GitHub Actions, 서비스형 자동매매, AI_Auto

단일 앱 구조의 한계

처음에는 Flask 대시보드와 엔진이 한 프로세스 안에서 같이 도는 구조가 이해하기는 쉬웠습니다. 하지만 이 방식은 서버가 떠 있지 않으면 실행이 멈추고, 환경이 바뀌면 로컬 파일과 SQLite에 저장하던 상태를 그대로 들고 가기 어렵다는 문제가 있었습니다.

특히 무료 또는 저비용 인프라로 옮기려면 상시 실행 프로세스를 억지로 붙잡는 것보다, 프론트와 저장소와 배치를 분리하는 것이 훨씬 현실적입니다. 그래서 아키텍처를 기능 중심이 아니라 책임 중심으로 다시 쪼개기 시작했습니다.

Vercel을 프론트와 서비스 콘솔로 제한한 이유

Vercel은 상시 엔진을 유지하는 곳보다는 운영 콘솔과 서버 라우트를 제공하는 데 강합니다. 그래서 현재 구조에서 Vercel은 개요, 모델 성과, 포지션, 설정 화면을 렌더링하고, 설정 화면에서 provider 키와 runtime profile을 Supabase에 저장하는 서비스 콘솔 역할만 맡도록 제한했습니다.

이 결정 덕분에 브라우저에서 비밀 키를 직접 쓰는 구조를 피할 수 있었고, 운영 입력은 Vercel 서버 함수를 통해서만 Supabase vault에 들어가게 만들 수 있었습니다. 즉 프론트는 보기 좋은 UI이면서 동시에 안전한 설정 입구가 되었습니다.

Supabase를 상태 원장으로 둔 이유

Supabase에는 heartbeat, model_setups, positions, daily_model_pnl, model_runtime_tunes, service_secrets 같은 운영용 원장을 두었습니다. 단순 대시보드 캐시가 아니라, 배치가 지나간 결과와 현재 상태가 무엇인지 다시 읽을 수 있는 기준 저장소를 마련한 것입니다.

또한 provider vault를 암호화해서 저장하는 기능까지 Supabase에 얹으면서, GitHub Secrets에 거래소 키를 직접 넣지 않아도 되는 구조를 만들었습니다. 이 차이는 개인 스크립트와 서비스형 구조를 가르는 핵심 포인트였습니다.

GitHub Actions를 배치 실행자로 택한 이유

지금 프로젝트는 실시간 틱 전략이 아니라 8분 배치형 선물 데모 구조로 운영하고 있습니다. 이런 구조라면 GitHub Actions가 주기 실행자로 충분히 역할을 할 수 있고, 하루 리포트 커밋과 주간 autotune도 같은 파이프라인 안에 묶을 수 있습니다.

물론 Actions는 정확히 초 단위 실행을 보장하지는 않지만, 현재 목표가 메이저 5개 코인 대상의 planner 기반 futures demo인 만큼, 운영 비용과 단순성을 생각하면 충분히 합리적인 선택입니다.

분리된 구조가 실제 운영에 주는 이점

Vercel, Supabase, GitHub Actions로 역할을 나누면 장애 원인도 훨씬 빨리 좁힐 수 있습니다. 화면이 안 뜨는 문제인지, 데이터 저장이 안 되는 문제인지, 배치가 멈춘 문제인지가 층별로 갈라지기 때문입니다. 단일 Flask 앱에 모든 것을 넣어두면 편해 보이지만, 한 군데에서 막히면 원인 탐색 자체가 길어집니다.

이번 프로젝트에서 이 구조를 선택한 이유도 여기에 있습니다. 서비스 콘솔은 운영자의 입력과 관측을 담당하고, Supabase는 상태 원장을 담당하고, Actions는 주기 실행을 담당하게 하면 각 레이어가 자기 책임만 가지게 됩니다. 작은 개인 프로젝트라도 이 구조는 운영 피로도를 크게 낮춰 줍니다.
