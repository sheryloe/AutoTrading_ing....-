Step 01

자동매매를 기능이 아니라 운영 리스크로 다시 정의한 이유

AI_Auto 프로젝트를 기능 추가 중심이 아니라 운영 리스크 관리 중심으로 다시 정의한 배경과 기준을 정리합니다.

게시일 2026-03-15 수정일 2026-03-15 키워드 자동매매, 운영 리스크, 트레이딩 시스템, 선물 자동매매, AI_Auto

문제를 새로 보는 기준

처음부터 다시 손본 이유는 모델 정확도가 낮아서가 아니라, 운영 화면과 상태 저장 방식이 실제 돈이 오가는 시스템의 기준을 충족하지 못했기 때문이었습니다. 한 화면 안에 신호, 포지션, 설정, 로그가 다 섞여 있으면 언뜻 많은 정보를 주는 것처럼 보이지만, 실제 운영에서는 무엇을 먼저 봐야 하는지 판단하기가 더 어려워집니다.

특히 선물 자동매매는 단순히 신호가 나오는지보다 상태가 어떻게 보존되고, 실수가 났을 때 어디까지 되돌릴 수 있으며, 다음 사이클에 무엇이 이어지는지가 훨씬 중요합니다. 그래서 이번 프로젝트는 기능을 늘리는 방향보다 운영 리스크를 줄이는 방향으로 우선순위를 완전히 뒤집었습니다.

운영자 시선에서 다시 쪼갠 구조

가장 먼저 정한 원칙은 입력과 관측을 분리하는 것이었습니다. 개요 화면은 지금 시스템이 살아 있는지와 최근 손익이 어떤지만 보여주고, 모델 성과 화면은 비교만 하며, 포지션 화면은 실제 실행 상태만 보여주고, 설정 화면만 운영 입력을 받도록 역할을 고정했습니다.

이 기준을 잡고 나니 화면을 예쁘게 꾸미는 문제보다 정보가 섞이지 않도록 레이어를 나누는 일이 먼저라는 점이 분명해졌습니다. 운영 리스크를 줄이는 첫 단계는 복잡한 데이터를 더 붙이는 것이 아니라, 어떤 데이터가 어느 맥락에서 보여야 하는지를 정리하는 일이었습니다.

문서가 운영 계약서가 되는 이유

자동매매 프로젝트는 코드만 좋다고 안정적으로 굴러가지 않습니다. 운영자가 어디서 키를 넣는지, 언제 리셋이 되는지, 하드 리셋을 하면 무엇이 지워지는지, 배치는 어느 주기로 도는지를 문서로 남겨 두지 않으면 결국 다음 변경 때 다시 같은 실수가 반복됩니다.

그래서 이번 정리에서는 README, GitHub Pages, GitHub Wiki, Notion까지 모두 같은 방향으로 문서를 맞추는 것을 기능 구현과 같은 레벨의 작업으로 봤습니다. 문서는 부가 설명이 아니라 운영 계약서에 가깝습니다. 화면과 배치와 저장소가 같은 언어를 쓰게 만드는 것이 곧 운영 품질입니다.

선물 시스템에서 더 엄격해야 하는 이유

이번 프로젝트는 spot이 아니라 futures를 기준으로 재구성하고 있습니다. 즉 진입 비중, 레버리지, 손절, 목표가, intrabar 체결 규칙이 모두 선물 가정 위에서 해석되어야 합니다. 과거의 데모 상태가 spot 성격의 가정 위에서 쌓여 있었다면, 그것을 그대로 이어받는 것 자체가 잘못된 비교가 됩니다.

바로 그 이유 때문에 단순히 화면만 고치는 것이 아니라, 선물 기준의 시드 정책과 하드 리셋 정책까지 같이 손보게 되었습니다. 운영 리스크라는 말은 거창하지만, 결국은 잘못된 전제를 끌고 다음 실험으로 넘어가지 않게 하는 장치입니다.

운영 리셋을 프로젝트 시작점으로 잡은 이유

운영 리스크를 먼저 정의하면 이후의 모든 의사결정이 달라집니다. 예를 들어 같은 지표를 추가하더라도 그 지표가 개요 화면에 보여야 하는지, 모델 성과 화면에 들어가야 하는지, 아니면 아예 설정 화면에서만 다뤄야 하는지 판단 기준이 생깁니다. 이번 프로젝트는 바로 그 기준을 먼저 세운 뒤에 나머지 기능을 붙였기 때문에, 보기 좋은 화면보다 운영 실수 가능성을 줄이는 화면이 우선순위가 되었습니다.

또 하나 중요한 점은 기록의 연속성입니다. 자동매매는 신호가 잘 나오는지보다 어떤 상태가 저장되고, 재시작 뒤에도 어떤 맥락이 이어지는지가 훨씬 중요합니다. 운영 리스크를 먼저 본다는 것은 결국 다음 사이클이 이전 사이클을 어떻게 이어받는지까지 책임지는 구조를 만든다는 뜻입니다.
