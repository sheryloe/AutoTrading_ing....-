Step 10

실거래로 가기 전에 꼭 확인해야 할 라이브 전환 체크리스트

현재 서비스형 futures demo 구조에서 실거래로 넘어가기 전에 무엇을 더 검증하고 고정해야 하는지 정리합니다.

게시일 2026-03-15 수정일 2026-03-15 키워드 실거래 전환, 라이브 가드, futures demo, 운영 체크리스트, AI_Auto

지금 상태는 어디까지 왔는가

현재 AI_Auto는 Vercel 서비스 콘솔, Supabase 상태 원장, GitHub Actions 배치, planner 기반 모델, 1분 intrabar 체결 시뮬레이션, 일별 PnL, 주간 autotune, 하드 리셋 흐름까지 갖춘 상태입니다. 다시 말해 화면과 배치와 상태 저장은 상당히 정리됐고, futures demo 운영을 반복적으로 검증할 수 있는 기반은 마련됐습니다.

하지만 여기서 곧바로 실거래로 넘어가는 것은 다른 문제입니다. 설정 화면에서 Bybit 키를 저장했다고 해서 live가 켜지지 않도록 만든 이유도, 이 전환이 별도의 검증 단계를 요구하기 때문입니다.

라이브 전환 전에 필요한 기술 검증

첫째는 order routing 자체를 별도 가드 아래에서 검증하는 일입니다. 지금 구조는 provider 저장과 execution target, live flags, arm 단계를 이미 나눠 두었지만, 실제 주문 라우팅은 아직 마지막 문을 열지 않은 상태입니다. 이 문은 체결 전 시뮬레이션과 잔고/포지션 동기화가 충분히 신뢰된 뒤에만 열어야 합니다.

둘째는 거래소 오류와 네트워크 지연에 대한 처리 방식을 정리하는 것입니다. 데모에서는 단순 실패로 끝날 수 있지만, live에서는 부분 체결, 중복 주문, 포지션 조회 지연 같은 운영 이슈가 곧 손실로 이어질 수 있습니다.

운영 체크리스트가 필요한 이유

자동매매 프로젝트에서 라이브 전환은 단순 기능 완성 체크가 아닙니다. 현재 키가 어떤 환경에 저장돼 있는지, arm이 켜져 있는지, execution target이 무엇인지, 최근 하드 리셋 이후 충분한 데모 성과가 쌓였는지, intrabar 충돌 규칙이 무엇인지까지 모두 확인해야 합니다.

즉 체크리스트는 문서 작업이 아니라 사고 방지 장치입니다. 운영자가 실거래 버튼을 눌렀을 때 어떤 전제를 믿고 있는지 명시하지 않으면, 문제가 생겼을 때 어디서부터 되짚어야 하는지조차 알기 어렵습니다.

지금 권장하는 운영 방식

지금은 futures demo를 한동안 더 굴리면서 모델별 체결 품질과 손익 곡선을 충분히 모으는 편이 맞습니다. 최소한 메이저 5종, 4개 모델, 8분 배치, intrabar 체결 규칙, 하드 리셋 이후 새 시드 기준 데이터가 한 주 이상 안정적으로 쌓이는지 보는 것이 좋습니다. 이 과정을 거쳐야 비로소 서비스형 자동매매라고 부를 수 있습니다.

라이브 전환 전에 더 필요한 검증

현재 구조는 실거래를 서두르기보다 라이브 전환 이전에 무엇을 더 고정해야 하는지 보여주는 단계에 가깝습니다. execution target, live flag, crypto live, arm 단계를 나눈 것도 같은 이유입니다. 키를 저장했다고 바로 주문이 나가면 운영자는 항상 불안한 상태에서 콘솔을 써야 하고, 작은 오해가 큰 사고로 이어질 수 있습니다.

그래서 이 단계에서 중요한 것은 기능을 더 붙이는 것이 아니라, 어떤 체크리스트를 통과했을 때만 다음 단계로 넘어갈지를 명확히 정하는 일입니다. provider 상태, 시뮬레이션 로그, 일별 손익, intrabar 판정, 하드 리셋 이후의 초기 상태가 모두 읽히는 상태여야만 그다음 라이브 라우팅을 열어도 됩니다.
