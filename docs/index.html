자세한 코드와 운영 문서는 GitHub 저장소 와 GitHub Wiki 에서 확인할 수 있습니다.

메이저 5종

유동성과 실행 안정성이 높은 심볼에 집중합니다.

8분

상시 루프 대신 운영 가능한 futures 배치 구조로 정리했습니다.

일별 PnL / 주간 튜닝

일별 손익 기록과 주간 autotune을 운영 루프에 포함했습니다.

서비스 콘솔

provider 키와 runtime profile을 운영 화면에서 분리 관리합니다.

왜 재구성했나

상태 확인과 설정 입력을 분리해야 운영 화면이 됩니다.

역할 분리

운영 상태 확인, 성과 비교, 포지션 점검, 설정 입력은 같은 순간에 같은 밀도로 보여줄 필요가 없습니다.

판단 속도

지표를 보다가 설정 폼을 만나는 구조는 판단을 느리게 만듭니다. 그래서 화면을 기능별로 분리했습니다.

제품 인상

GitHub Pages는 프로젝트 소개와 진입점에 집중하고, 실제 운영은 콘솔과 Wiki 문서로 분리했습니다.

화면 구조

운영자 시선에 맞춘 4개 화면 구조

01 개요

핵심 KPI, heartbeat, 최근 사이클 요약, 오픈 포지션 수만 보여줍니다.

02 모델 성과

모델별 PnL, 승률, 종료 거래 수, autotune 상태를 비교합니다.

03 포지션

오픈 포지션, 최신 setup, entry / SL / TP 상태를 실행 기준으로 모읍니다.

04 설정

Service control, provider vault, execution target, runtime profile은 설정 화면에만 둡니다.

화면 분위기

하이퍼리퀴드 계열의 다크 운영 도구 인상

검은 배경, 민트와 시안 포인트, 데이터 패널 중심 레이아웃을 사용했습니다. 과한 소개 문구보다 운영 도구처럼 보이는 질감을 우선했습니다.

  • 페이지별 책임 분리
  • 데이터 카드와 표 중심 레이아웃
  • 설정 입력은 별도 화면으로 격리

모델 운영

4개 planner 모델이 진입과 청산 계획을 제안합니다.

A 레인지 리버전

되돌림 구간과 평균 회귀를 중심으로 entry와 stop 구조를 계산합니다.

B 리클레임

스윕 후 회복되는 패턴에서 reclaim 기준의 진입 계획을 만듭니다.

C 압축 돌파

변동성 수축 이후 breakout retest를 기준으로 목표 구간을 설정합니다.

D 리셋 바운스

실패한 돌파 이후 반대 방향으로 복귀하는 구간을 활용합니다.

배포 구조

프론트, 저장소, 배치를 분리해 운영합니다.

Vercel

운영자 콘솔과 Service control 화면을 제공합니다.

Supabase

상태 저장, runtime profile, provider vault, 대시보드 원장을 담당합니다.

Python 배치

8분 분석, intrabar 체결 판정, 일별 리포트, 주간 autotune 작업을 처리합니다.

참고
